🚀 LIMBO GAME SPEED OPTIMIZATION PLAN
=========================================

Goal:
-----
Reduce delay between placing a bet and showing results — users should see outcomes instantly (<100ms) while blockchain operations run asynchronously.

------------------------------------------------
1️⃣ CORE STRATEGY
------------------------------------------------

✅ Use Off-chain Instant Result Generation
- Compute provably fair random results on backend instantly (no Chainlink delay).
- Send result to frontend immediately with verification data.
- Continue blockchain operations in background (async).

✅ Optimistic UI Updates
- Deduct bet amount from displayed balance immediately.
- Update with win/loss instantly.
- Sync periodically to reflect verified on-chain or DB balance.

✅ Asynchronous Blockchain Handling
- Never wait for on-chain confirmations in user response.
- Use background worker or queue to process transactions (e.g., house wallet payouts).

------------------------------------------------
2️⃣ NEXT.JS BACKEND OPTIMIZATION
------------------------------------------------

🔹 Implement async tasks:
  - Use `processBlockchainTransactions()` as detached promise (no await).
  - Optionally move to background queue (BullMQ / Upstash QStash / custom worker).

🔹 Use batched DB writes:
  - Group bet + transaction inserts in a single Prisma transaction.

🔹 Preload and cache ETH price:
  - Refresh every 10 seconds; no external API hit on each bet.

🔹 Minimize encryption overhead:
  - Cache decrypted house wallet once per process (securely).

------------------------------------------------
3️⃣ FRONTEND OPTIMIZATION
------------------------------------------------

🔹 Instant UX update flow:
  1. Deduct user balance instantly.
  2. Show animation → “Rolling…” for 1s.
  3. Show win/loss immediately after backend response.
  4. Push backend response to local state instantly.
  5. Schedule silent sync to `/api/user/balance` in 5s.

🔹 Periodic background balance refresh:
  - Use setInterval every 5–10s.
  - Or use WebSocket to receive balance/payout updates in real-time.

🔹 Lazy load user data:
  - Load critical data (balance, active bets) first.
  - Lazy load history & verification logs later.

------------------------------------------------
4️⃣ DATABASE & SERVER TUNING
------------------------------------------------

✅ Use connection pooling (e.g., Prisma with pgbouncer).
✅ Run PostgreSQL on SSD-backed instance.
✅ Enable Prisma query caching where applicable.
✅ Use indexes on:
  - `bets.userId`
  - `bets.status`
  - `walletTransactions.userId`

------------------------------------------------
5️⃣ OPTIONAL PERFORMANCE FEATURES
------------------------------------------------

💡 WebSocket push:
   - Push payout updates instantly instead of polling.

💡 Redis cache layer:
   - Cache user balance, ETH/USD rate, and recent bets.

💡 Queue/Worker system:
   - Offload blockchain transactions to separate Node worker.

💡 Serverless function warm-up:
   - Keep the `/api/bet` route warm to avoid cold start delays.

------------------------------------------------
6️⃣ TARGET PERFORMANCE METRICS
------------------------------------------------

Operation | Expected Time | Type
-----------|----------------|-----
Bet placement → Result shown | <100ms | Synchronous (frontend)
Async blockchain processing | 5–15s | Background
Balance reconciliation | 1–2s | Auto-sync
Verification display | Instant | Cached JSON

------------------------------------------------
✅ SUMMARY
------------------------------------------------

Frontend: Instant feedback, smooth animations, optimistic updates.
Backend: Off-chain provably fair results, async payouts, batched DB writes.
Blockchain: Detached async flow with queued confirmation.
Users experience a “real-time casino” feel — results appear instantly, with background-secured blockchain integrity.
